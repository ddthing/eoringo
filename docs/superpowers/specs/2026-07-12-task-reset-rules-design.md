# 숙제별 리셋 규칙 데이터 구조 설계

## 목표

숙제마다 서로 다른 초기화 시각과 주기를 한국 표준시 기준으로 정확하게 처리한다. 기존 완료 데이터, History, 백업을 보존하며 향후 다음 초기화 표시와 리셋 주기별 관리 UI에서 재사용할 수 있는 순수 데이터 구조를 만든다.

이번 작업은 리셋 데이터 기반 구축까지 포함한다. 숙제 검색, 필터, 접기 등 관리 UI 개편은 별도 설계로 진행한다.

## 핵심 원칙

- 모든 고정 시각은 `Asia/Seoul` 기준이다.
- 화면 분류인 `category`와 실제 초기화 규칙을 분리한다.
- `TaskTemplate`은 실행 로직 대신 직렬화 가능한 `resetRuleId`를 참조한다.
- 리셋 계산기는 Zustand와 React에 의존하지 않는 순수 함수로 구현한다.
- 기존 저장 데이터는 v6 마이그레이션으로 자동 변환한다.
- 한 단계의 테스트를 통과하기 전에는 다음 단계로 넘어가지 않는다.

## 리셋 규칙 모델

### 규칙 종류

```ts
type ResetRule =
  | { kind: "daily"; hour: number; minute: number; timeZone: "Asia/Seoul" }
  | { kind: "weekly"; weekday: number; hour: number; minute: number; timeZone: "Asia/Seoul" }
  | { kind: "interval"; durationHours: number }
  | { kind: "manual" };
```

고정 규칙은 레지스트리에 정의한다.

- `daily-midnight`: 매일 00:00
- `daily-0500`: 매일 05:00
- `daily-1700`: 매일 17:00
- `weekly-tue-1700`: 매주 화요일 17:00
- `weekly-sat-2100`: 매주 토요일 21:00
- `interval-18h`: 완료 후 18시간
- `manual`: 자동 초기화 없음

### TaskTemplate

- `resetRuleId`를 필수 필드로 추가한다.
- 기존 `resetType`은 호환 레이어에서 읽을 수 있도록 유지하되 새 로직의 기준으로 사용하지 않는다.
- `category`는 오늘/주간/기타 진행도와 화면 탭 분류에만 사용한다.
- 선택적 `availabilityRuleId`와 `retentionDays` 메타데이터를 허용한다.

## 숙제별 규칙 매핑

### 매일 00:00

- 무작위 임무 전체
- 우호부족 일일 퀘스트
- 골드소서 일일 항목

### 매일 05:00

- 총사령부 조달 납품
- 총사령부 군수품 납품

### 매일 17:00

- 무인도 목장

### 매주 화요일 17:00

- 단골 거래
- 주간 석판 목표
- 환 토벌전
- 24인 레이드 주간 보상
- 8인 레이드 주간 보상
- 청마도사 가면 무투회
- 무인도 공방
- 공략수첩
- 총사령부 소대 중요 임무
- 패션 체크
- 쿠로의 공상수첩

### 매주 토요일 21:00

- 골드소서 주간 항목

### 완료 후 18시간

- 집사 수행
- 보물지도 채집
- 총사령부 소대 일반 임무

### 추가 메타데이터

- 패션 체크는 화요일 17시에 초기화하고 금요일 17시 참여 가능 규칙을 별도 저장한다.
- 쿠로의 공상수첩은 `retentionDays: 14`를 저장하지만 자동 만료시키지 않는다. 향후 수령일 입력 기능이 생길 때 사용한다.

## 기본 숙제 분리

### 무인도

기존 `daily-island`를 다음 두 항목으로 대체한다.

- `daily-island-pasture`: 무인도 목장, 매일 17:00
- `weekly-island-workshop`: 무인도 공방, 매주 화요일 17:00

기존 `weekly-island` 항목은 새 공방 항목으로 정규화한다.

### 총사령부 소대

기존 `grand-company-squadron-weekly`를 다음 두 항목으로 대체한다.

- `grand-company-squadron-routine`: 소대 일반 임무, 완료 후 18시간
- `grand-company-squadron-priority`: 소대 중요 임무, 매주 화요일 17:00

## 스토어 구조

기존 전역 `dailyResetKey`와 `weeklyResetKey` 대신 다음 상태를 사용한다.

```ts
resetKeysByRule: Record<ResetRuleId, string>
completedAtByCharacter: Record<ScopeId, Record<TaskId, string>>
```

- 고정 규칙은 현재 시점이 속한 리셋 주기의 안정적인 키를 계산한다.
- 저장된 규칙 키와 현재 키가 달라진 규칙에 속한 완료 상태만 제거한다.
- 18시간 규칙은 최초 완료 시각을 기록하고 18시간이 지난 항목만 제거한다.
- 수동 규칙은 자동으로 제거하지 않는다.
- 알 수 없는 규칙 ID는 오류를 던지지 않고 `manual`처럼 보존한다.

## v6 마이그레이션

### 리셋 키

- 기존 `dailyResetKey`는 `daily-midnight`의 초기 기준으로 변환한다.
- 기존 `weeklyResetKey`는 과거 형식이 화요일 자정이므로 새 화요일 17시 키로 직접 재사용하지 않는다.
- 마이그레이션 시점의 현재 규칙 키를 초기값으로 기록해 앱 업데이트 직후 완료 상태가 예기치 않게 전부 삭제되지 않게 한다.

### 분리 항목 완료 상태

- `daily-island` 완료 상태를 목장과 공방에 모두 승계한다.
- `weekly-island` 완료 상태를 공방에 승계하고 기존 `daily-island` 승계값과 병합한다.
- `grand-company-squadron-weekly` 완료 상태를 소대 일반 임무와 중요 임무에 모두 승계한다.
- 기존 완료 시각이 있으면 18시간 항목에 승계한다. 없으면 마이그레이션 시각을 완료 시각으로 사용한다.
- 기존 ID는 완료 및 완료 시각 맵에서 제거한다.

### 활성/비활성 설정

- 기존 무인도 항목 중 하나라도 비활성화되어 있으면 대응되는 새 항목에 승계한다.
- 기존 소대 항목의 비활성 상태는 두 새 항목에 모두 승계한다.
- 중복 ID는 `Set`으로 제거한다.

### 커스텀 숙제

- 기존 `daily`는 `daily-midnight`로 변환한다.
- 기존 `weekly`는 `weekly-tue-1700`으로 변환한다.
- 기존 `eighteenHours`는 `interval-18h`로 변환한다.
- 기존 `manual` 및 알 수 없는 값은 `manual`로 정규화한다.

## 수주권 데이터

수주권은 완료 여부를 기록하는 숙제가 아니라 누적 자원이므로 별도 Zustand 스토어로 관리한다.

```ts
type AllowanceState = {
  value: number;
  lastAccrualKey: string;
};
```

- 최초 값은 0이다.
- KST 09:00과 21:00마다 3장 증가한다.
- 최대 100장이다.
- 앱을 오래 열지 않았으면 마지막 누적 키 이후 지난 회차 수를 계산해 한 번에 반영한다.
- 사용자는 현재 수량을 직접 줄이거나 수정할 수 있다.
- 미래 시각이나 손상된 값은 0~100 범위와 현재 누적 키로 정규화한다.
- 이번 데이터 단계에서는 별도 화면을 추가하지 않는다.

## History와 백업

- History의 `HistoryTask.resetType`은 기존 백업 판독을 위해 유지하고 `resetRuleId`를 추가한다.
- 날짜별 History는 현재처럼 캘린더 날짜 스냅샷을 유지한다.
- 고정 시각 리셋 직전 상태를 별도 이벤트로 저장하는 기능은 이번 범위에 포함하지 않는다.
- 백업 버전은 새 필드와 수주권 스토어를 포함하도록 올린다.
- 이전 버전 백업 복원 시 누락된 규칙은 v6 정규화 로직으로 채운다.
- 새 백업을 이전 앱에서 읽는 역호환은 보장하지 않는다.

## 오류 처리

- 잘못된 시각, 요일, 기간을 가진 규칙은 등록 단계에서 거부한다.
- 저장 데이터의 알 수 없는 규칙은 수동 규칙으로 처리하여 완료 데이터를 보존한다.
- 잘못된 ISO 완료 시각은 만료로 간주하지 않고 정규화 과정에서 제거한다.
- 리셋 동기화는 동일 시각에 여러 번 실행해도 결과가 같은 멱등 연산이어야 한다.
- DST가 없는 KST 기준이지만 계산은 명시적 시간대 변환을 사용한다.

## 단계별 구현과 검증 게이트

### 1단계: 순수 규칙 계산기

- 일일·주간 경계 직전/정각/직후 테스트
- 월·연도 경계 테스트
- 18시간 만료 테스트
- 잘못된 규칙 테스트

### 2단계: 기본 숙제 매핑

- 모든 기본 숙제에 유효한 `resetRuleId`가 있는지 테스트
- 분리된 ID와 제목, 범주, 규칙 매핑 테스트

### 3단계: 스토어 v6 마이그레이션

- 기존 완료/비활성 상태 승계 테스트
- 규칙별 선택 초기화 테스트
- 앱 업데이트 직후 일괄 삭제 방지 테스트

### 4단계: 수주권

- 09:00·21:00 경계 테스트
- 여러 회차 누적, 100장 상한, 수동 수정 테스트
- 손상 데이터 정규화 테스트

### 5단계: History·백업

- 이전 백업 복원 테스트
- 새 규칙과 수주권 왕복 테스트
- 기존 History 데이터 정규화 테스트

### 6단계: 최종 검증

- 전체 테스트
- TypeScript 프로덕션 빌드
- 기존 숙제 체크·카운트·캐릭터 전환 회귀 확인

## 제외 범위

- 숙제 관리 검색, 필터, 접기 UI
- 다음 초기화 시각 표시 UI
- 리셋 알림
- 수주권 표시 및 편집 UI
- 쿠로 수령일 입력과 자동 만료
- 리셋 직전 History 이벤트 기록
